Prep, Prerequisites and the Health Endpoint

▼

Initial preparation phase including environment setup, dependency verification, and health endpoint implementation. Ensures the application is ready for containerization and deployment to EKS.

✅ Services: Pay247CppCommon, promocodews

✅ Module Keys: pay247cppcommon, promocodews

✅ Steps:

  • Ensure proper pom.xml plugins: 
    <plugin>
  <artifactId>maven-war-plugin</artifactId>
  <version>3.3.2</version>
</plugin>
<plugin>
  <artifactId>maven-compiler-plugin</artifactId>
  <version>3.10.1</version>
</plugin>
  • Add a health endpoint to Pay247CppCommon and Promocodews: 
    @RequestMapping(path = "/health", method = RequestMethod.GET)
public String health() throws UnknownHostException {
    return "{\"application\": {\"name\": \"<SERVICE-NAME>\", \"lbstatus\": \"in\"}}";
}
  • UI services: add src/main/webapp/health/index.html: 
    <H1>Service is running</H1>
  • For SOAP API services, add servlet configuration in web.xml: 
    <servlet>
    <servlet-name>HealthServlet</servlet-name>
    <servlet-class>edu.apollogrp.financials.service.HealthServlet</servlet-class>
</servlet>
<servlet-mapping>
    <servlet-name>HealthServlet</servlet-name>
    <url-pattern>/health</url-pattern>
</servlet-mapping>
  • Create HealthServlet.java class: 
    public class HealthServlet extends HttpServlet {
    protected void doGet(HttpServletRequest request, HttpServletResponse response)
            throws ServletException, IOException {
        response.setStatus(HttpServletResponse.SC_OK);
        response.setContentType("application/json");
        response.getWriter().write("{\"service\":\"pay247_cppcommon\", \"status\":\"OK\"}");
    }
                                                                                 }
  • Added Tomcat Servlet API dependency to pom.xml to support health endpoint in SOAP API services: 
    <dependency>
    <groupId>org.apache.tomcat</groupId>
    <artifactId>tomcat-servlet-api</artifactId>
    <version>8.0.32</version>
</dependency>

    • Note: Similar implementation for promocodews service with appropriate service name.

MSK Migration - Moving from the from Oracle Service Bus AKA Middleware

🏢

Company-Wide Initiative

Strategic move to modernize event streaming infrastructure and reduce dependency on legacy OSB platform

🔧

Gradual OSB Decomposition

Consumer-by-consumer migration approach allows systematic breakdown of monolithic OSB architecture

💰

Financial Event Processing

Both CM (Credit Memo) and CI (Credit Invoice) are critical financial events requiring reliable, scalable processing

⚡

Real-Time Capabilities

MSK provides sub-second event delivery and stream processing for immediate financial data insights

🔄

Zero-Disruption Migration

Dual-channel publishing ensures existing OSB consumers continue working while new MSK consumers are onboarded

Oracle Queue-Based Event Staging for MSK Integration

🔄

Event Routing Control

Oracle lookup values control event publishing between OSB middleware and Kafka infrastructure using XXUOP_EVENTS_PUBLISHER_STATUS

OSB - Oracle Service Bus only KAFKA - Oracle AQ staging only ALL - Both OSB and Kafka
📋

Credit Invoice Queue

Queue: XXUOP_CI_EVENT_Q

Object Type: XXUOP_CI_EVENT_OBJ

Configuration: 5 retries, 60s delays, 7-day retention

💰

Credit Memo Queue

Queue: XXUOP_CM_EVENT_Q

Object Type: XXUOP_CM_EVENT_OBJ

Configuration: 5 retries, 60s delays, 7-day retention

🏗️

Queue Infrastructure Benefits

Reliable message delivery with automatic retry mechanisms, comprehensive error handling, and configurable retention policies for audit compliance

⚙️

Seamless Integration

Bridge between legacy Oracle systems and modern Kafka infrastructure, enabling gradual migration with zero disruption to existing workflows

MSK Integration Bridge & Migration Strategy

MSK Integration Bridge

🐳

ECS Service: st-ci

Polls XXUOP_CI_EVENT_Q every 500ms

Publishes to MW.CUSTOMER.FINANCIALS.CI_EVENT

AND
🐳

ECS Service: st-cm

Polls XXUOP_CM_EVENT_Q every 500ms

Publishes to MW.CUSTOMER.FINANCIALS.CM_EVENT

Zero-Downtime Migration Strategy

1

Initial Phase

Set lookup to ALL - events flow to both OSB and Kafka

2

Migration Phase

Gradually migrate consumers from OSB to Kafka

3

Completion Phase

Set lookup to KAFKA for Kafka-only publishing
